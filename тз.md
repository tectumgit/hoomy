# 3. Техническое задание (ТЗ)

## Главный документ разработки мобильного приложения «Хоми»

---

# 1. Назначение проекта

**Хоми** — мобильное приложение для покупки продуктов и товаров большими объемами напрямую у поставщиков с плановой доставкой.

Приложение соединяет:

**Клиент → Поставщик**

Без представителей внутри приложения.

Главная особенность проекта:

* клиент покупает товары крупными объемами;
* у каждого товара есть минимальный объем заказа;
* поставщик сам размещает товары;
* товары автоматически публикуются в каталоге;
* поставщик сам выбирает зоны доставки;
* поставщик сам задает стоимость доставки;
* доставка не экспресс, а плановая;
* клиент может покупать товары от нескольких поставщиков;
* корзина разделяется по поставщикам;
* клиент может напрямую писать поставщику;
* при споре подключается поддержка Хоми.

---

# 2. Цель разработки

Создать MVP мобильного приложения и web-системы, которые позволяют:

* клиенту зарегистрироваться по SMS;
* выбрать товары в каталоге;
* увидеть минимальный объем товара;
* добавить товары от разных поставщиков в корзину;
* оформить заказ;
* оплатить заказ;
* выбрать дату доставки;
* напрямую связаться с поставщиком;
* подать жалобу или открыть спор;
* поставщику зарегистрироваться по SMS;
* поставщику оформить публичный профиль;
* поставщику добавить товары;
* поставщику управлять зонами и стоимостью доставки;
* поставщику получать и обрабатывать заказы;
* администратору проверять поставщиков, товары, жалобы и споры.

# 4. Роли пользователей

## 4.1. Клиент

Клиент — пользователь, который покупает товары.

Клиент может:

* зарегистрироваться по номеру телефона;
* войти по SMS-коду;
* смотреть каталог;

* покупать товары от нескольких поставщиков;
* выбирать количество товара;

* открывать спор;

* войти по SMS-коду;

* загрузить обложку;
* добавить описание о себе;

* указать минимальный объем товара;
* указать единицу измерения;
* указать шаг заказа;
* указать остаток;
* указать даты доставки;
* выбрать зоны или регионы доставки;
* задать стоимость доставки;
* задать бесплатную доставку от суммы;
* получать заказы;
* менять статусы заказов;
* общаться с клиентом напрямую;
* видеть аналитику продаж и просмотров.

## 4.3. Администратор

Администратор — сотрудник Хоми.

Администратор может:

* проверять поставщиков;
* одобрять или отклонять поставщиков;
* просматривать товары;
* скрывать товары;
* блокировать товары;
* управлять категориями;
* управлять заказами;
* видеть условия доставки поставщика;
* рассматривать жалобы;
* подключаться к спорам;
* принимать решения по спорам;
* блокировать клиентов;
* блокировать поставщиков;
* смотреть аналитику платформы.

Администратор не должен:

* вручную одобрять каждый товар перед публикацией;
* задавать стоимость доставки за поставщика;
* управлять доставкой вместо поставщика.

# 5. Авторизация и регистрация

## 5.1. Авторизация по SMS

Вход в систему для клиента и поставщика должен выполняться через:

* номер телефона;
* SMS-код.

---

## 5.2. Регистрация клиента

Клиент вводит:

* номер телефона;
* SMS-код.

После первого входа клиент заполняет:

* имя;
* город;
* адрес доставки.

Email — необязательное поле.

---

## 5.3. Регистрация поставщика

Поставщик вводит:

* номер телефона;
* SMS-код.

После входа поставщик заполняет профиль.

### Обязательные поля поставщика:

* название поставщика;
* ИНН;
* ОГРН / ОГРНИП;
* контактное лицо;
* телефон;
* email;
* город;
* зоны доставки;
* категории товаров.

### Для публичного профиля:

* логотип;
* обложка;
* описание о себе;
* условия доставки.

---

## 5.4. Что не требуется у поставщика в MVP

Не делать обязательными:

* адрес склада;
* адрес точки отгрузки;
* срок годности по каждому товару.

Адрес склада можно оставить только как внутреннее необязательное поле для администрации на будущих этапах.

# 6. Основная бизнес-логика

## 6.1. Минимальный объем товара

Каждый товар должен иметь минимальный объем заказа.

Поставщик при добавлении товара указывает:

* единицу измерения;
* минимальный объем;
* шаг заказа;
* цену за единицу;
* доступный остаток;
* дату доставки.

---

## 6.2. Единицы измерения

В системе должны быть доступны следующие единицы измерения:

| Единица  | Пример                       |
| -------- | ---------------------------- |
| кг       | картофель, мясо, сахар, мука |
| г        | специи, сыр, деликатесы      |
| л        | молоко, масло, вода          |
| мл       | соусы, напитки               |
| шт       | яйца, упаковки               |
| упаковка | бытовая химия, крупы         |
| коробка  | напитки, консервы            |
| ящик     | овощи, фрукты                |
| мешок    | мука, сахар, крупы           |
| набор    | фермерские наборы            |

---

## 6.3. Минимальный объем и шаг заказа

Важно разделять:

### Минимальный объем

Минимальное количество товара, которое можно купить.

Пример:

```text
Минимальный объем: 10 кг
```

### Шаг заказа

Количество, на которое клиент может увеличивать заказ.

Пример:

```text
Шаг заказа: 5 кг
```

Тогда клиент может выбрать:

```text
10 кг, 15 кг, 20 кг, 25 кг, 30 кг
```

Клиент не может выбрать:

```text
1 кг, 2 кг, 7 кг, 12 кг
```

---

## 6.4. Формула расчета стоимости товара

```text
Стоимость товара = цена за единицу × количество
```

Пример:

```text
Картофель: 38 ₽ × 10 кг = 380 ₽
Молоко: 90 ₽ × 6 л = 540 ₽
Яйца: 10 ₽ × 30 шт = 300 ₽
```

---

## 6.5. Минимальная сумма заказа у поставщика

Кроме минимального объема товара, у поставщика может быть минимальная сумма заказа.

Пример:

```text
Минимальная сумма заказа у поставщика: 3 000 ₽
```

Если клиент добавил товаров на 2 400 ₽, система показывает:

```text
До минимального заказа осталось 600 ₽
```

---

## 6.6. Автоматическая публикация товаров

Товары не проходят предварительную ручную модерацию.

Логика:

* поставщик создает товар;
* нажимает «Опубликовать»;
* товар автоматически появляется в каталоге;
* администратор может позже скрыть или заблокировать товар при нарушении.

# 7. Функционал клиента

## 7.1. Главный экран

На главном экране должны быть:

* поиск;
* категории товаров;
* популярные товары;
* акции;
* ближайшие даты доставки;
* подборки;
* повтор последнего заказа.

Примеры подборок:

* «Овощи коробками»;
* «Закупка на неделю»;
* «Товары для семьи»;
* «Фермерские продукты»;
* «Для кафе и офисов».

---

## 7.2. Каталог товаров

Каталог должен содержать:

* категории;
* подкатегории;
* список товаров;
* фильтры;
* сортировку.

### Фильтры:

* категория;
* цена;
* поставщик;
* ближайшая дата доставки поставщика;
* зона доставки;
* минимальный объем;
* единица измерения;
* наличие;
* рейтинг поставщика.

### Сортировка:

* по цене;
* по популярности;
* по ближайшей дате доставки поставщика;
* по рейтингу;
* по размеру минимального заказа.

---

## 7.3. Публичный профиль поставщика

Клиент должен видеть страницу поставщика.

Профиль поставщика содержит:

* логотип;
* обложку;
* название;
* описание;
* город;
* зоны доставки;
* условия доставки;
* категории товаров;
* рейтинг;
* количество отзывов;
* каталог товаров поставщика;
* кнопку «Написать поставщику».

---

## 7.4. Карточка товара

Карточка товара должна содержать:

* фото;
* название;
* описание;
* поставщик;
* рейтинг поставщика;
* цена за единицу;
* единица измерения;
* минимальный объем;
* шаг заказа;
* доступный остаток;
* итоговая цена;
* условия хранения — опционально;
* кнопка «Добавить в корзину»;
* кнопка «Написать поставщику».

### Не указывать в карточке товара:

* срок годности как обязательное поле;
* адрес склада;
* адрес точки отгрузки;
* дата и стоимость доставки как свойства товара.

---

## 7.5. Выбор количества

Клиент может увеличивать количество только по правилам товара.

Пример:

```text
Минимум: 10 кг
Шаг: 5 кг
```

Доступные значения:

```text
10, 15, 20, 25, 30 кг
```

Если клиент пытается выбрать меньше минимума, система показывает:

```text
Минимальный заказ этого товара — 10 кг
```

Если клиент вводит неправильный шаг, система показывает:

```text
Количество можно увеличивать только с шагом 5 кг
```

---

## 7.6. Корзина от нескольких поставщиков

Корзина должна поддерживать товары от нескольких поставщиков.

Если клиент добавил товары от разных поставщиков, корзина разделяется по поставщикам.

Рекомендуемый интерфейс:

```text
Корзина
├── Поставщик 1
├── Поставщик 2
├── Поставщик 3
```

Можно реализовать в виде вкладок, как в Яндекс Еде.

---

## 7.7. Содержимое вкладки поставщика в корзине

В каждой вкладке поставщика показывается:

* название поставщика;
* товары этого поставщика;
* количество;
* цена за единицу;
* сумма по каждому товару;
* минимальная сумма заказа у поставщика;
* стоимость доставки поставщика;
* бесплатная доставка от суммы, если есть;
* дата доставки;
* итоговая сумма по поставщику.

---

## 7.8. Общий итог корзины

Общая корзина показывает:

* итог по всем поставщикам;
* общую сумму товаров;
* общую сумму доставок;
* итог к оплате.

---

## 7.9. Пример корзины

| Поставщик     | Товары          | Сумма товаров | Доставка |
| ------------- | --------------- | ------------: | -------: |
| Фермер Казань | картофель, яйца |       2 800 ₽ |    300 ₽ |
| Молочная база | молоко, сыр     |       3 500 ₽ |    250 ₽ |
| ОптОвощ       | яблоки, морковь |       2 200 ₽ |    350 ₽ |

Итог:

```text
Товары: 8 500 ₽
Доставка: 900 ₽
К оплате: 9 400 ₽
```

---

## 7.10. Оформление заказа

Клиент указывает:

* адрес доставки;
* дату доставки;
* контактный телефон;
* комментарий к заказу;
* способ оплаты.

Перед оплатой клиент видит:

* товары;
* поставщиков;
* количество;
* цену;
* доставку по каждому поставщику;
* итоговую сумму;
* дату доставки.

---

## 7.11. Оплата

В MVP нужна онлайн-оплата.

Способы оплаты:

* банковская карта;
* СБП — желательно.

Оплата для юридических лиц — позже.

После оплаты заказ получает статус:

```text
Оплачен
```

---

## 7.12. Статусы заказа для клиента

| Статус             | Значение                      |
| ------------------ | ----------------------------- |
| Создан             | заказ оформлен, но не оплачен |
| Оплачен            | деньги получены               |
| Принят поставщиком | поставщик подтвердил заказ    |
| В сборке           | поставщик собирает заказ      |
| Передан в доставку | заказ передан в доставку      |
| В пути             | заказ едет к клиенту          |
| Доставлен          | заказ получен клиентом        |
| Отменен            | заказ отменен                 |
| Возврат            | оформлен возврат              |
| Спор               | по заказу открыт спор         |

---

## 7.13. История заказов

Клиент может:

* смотреть прошлые заказы;
* повторить заказ;
* скачать чек;
* написать поставщику;
* оставить отзыв;
* открыть спор;
* обратиться в поддержку Хоми.

---

## 7.14. Отзывы и рейтинг

Клиент может оценить:

* товар;
* поставщика;
* доставку.

Оценка:

* от 1 до 5.

Комментарий:

* текст;
* фото — опционально.

# 8. Коммуникация клиента и поставщика

## 8.1. Прямой чат

В приложении должен быть чат между клиентом и поставщиком.

Клиент может написать поставщику:

* до заказа;
* после заказа;
* по конкретному товару;
* по конкретному заказу.

Поставщик может отвечать клиенту.

---

## 8.2. Что должно быть в чате

* текстовые сообщения;
* история переписки;
* привязка к заказу или товару;
* уведомления о новых сообщениях;
* возможность прикрепить фото — опционально.

---

## 8.3. Жалоба на поставщика

Клиент может подать жалобу на поставщика.

Причины жалобы:

* товар не соответствует описанию;
* товар плохого качества;
* поставщик не отвечает;
* доставка сорвана;
* неверная цена;
* грубое общение;
* другая причина.

---

## 8.4. Спор по заказу

Если клиент и поставщик не решили вопрос напрямую, клиент может открыть спор.

После открытия спора подключается поддержка Хоми.

---

## 8.5. Статусы спора

| Статус                    | Значение                               |
| ------------------------- | -------------------------------------- |
| Создан                    | клиент открыл спор                     |
| На рассмотрении           | поддержка Хоми изучает обращение       |
| Запрошена информация      | нужны данные от клиента или поставщика |
| Решен в пользу клиента    | поддержка приняла сторону клиента      |
| Решен в пользу поставщика | поддержка приняла сторону поставщика   |
| Закрыт                    | спор завершен                          |

---

## 8.6. Администратор в споре

Администратор может:

* просмотреть заказ;
* просмотреть переписку;
* запросить информацию;
* принять решение;
* оформить возврат;
* закрыть спор;
* вынести предупреждение поставщику;
* скрыть товары;
* заблокировать поставщика.

---

# 9. Функционал поставщика

## 9.1. Верификация поставщика

До проверки поставщик может создать профиль, но не должен иметь полный доступ к продажам.

Статусы поставщика:

| Статус       | Значение                  |
| ------------ | ------------------------- |
| На проверке  | поставщик отправил данные |
| Одобрен      | может продавать           |
| Отклонен     | не прошел проверку        |
| Заблокирован | доступ ограничен          |

---

## 9.2. Кабинет поставщика

Поставщик должен видеть:

* профиль;
* товары;
* заказы;
* продажи;
* остатки;
* зоны доставки;
* условия доставки;
* выплаты;
* рейтинг;
* отзывы;
* сообщения клиентов;
* жалобы;
* аналитику.

---

## 9.3. Публичный профиль поставщика

Поставщик загружает:

* логотип;
* обложку;
* название;
* описание о себе;
* город;
* зоны доставки;
* категории товаров;
* условия доставки.

Клиент видит эту информацию в приложении.

---

## 9.4. Добавление товара

Поставщик указывает:

* название товара;
* категорию;
* фото;
* описание;
* цену;
* единицу измерения;
* минимальный объем;
* шаг заказа;
* доступный остаток;
* статус товара.

### Необязательное поле:

* условия хранения.

### Не нужно указывать:

* срок годности;
* адрес склада;
* адрес точки отгрузки;
* дату доставки;
* стоимость доставки;

# 11. Доставка

## 11.1. Тип доставки

В Хоми используется **плановая доставка**.

Это означает:

* не срочная доставка;
* не экспресс-доставка;
* доставка по доступным датам и зонам поставщика.

---

## 11.2. Кто управляет доставкой

Доставкой управляет поставщик.

Поставщик сам задает:

* зоны доставки;
* регионы доставки;
* дни доставки;
* временные интервалы;
* стоимость доставки;
* бесплатную доставку от суммы;
* минимальную сумму заказа.

---

## 11.3. Даты доставки

Поставщик указывает доступные даты доставки.

Пример:

```text
Доставка: вторник, четверг, суббота
```

Клиент выбирает доступную дату при оформлении заказа.

---

## 11.4. Дедлайн заказа

Для каждой даты доставки должен быть дедлайн оформления заказа.

Пример:

```text
Заказ до 18:00 сегодня — доставка завтра.
```

Если дедлайн прошел, дата становится недоступной.

---

## 11.5. Стоимость доставки

Стоимость доставки указывает поставщик.

Варианты:

* фиксированная цена;
* доставка по зонам;
* доставка по району;
* бесплатная доставка от суммы;
* разные условия по городам или регионам.

---

## 11.6. Администратор и доставка

Администратор может:

* видеть условия доставки;
* скрыть поставщика при нарушениях;
* заблокировать некорректные условия;
* рассматривать жалобы на доставку.

Администратор не должен:

* задавать стоимость доставки;
* менять зоны доставки вместо поставщика;
* управлять графиком доставки вместо поставщика.

---

# 12. Возвраты, отмены, жалобы

## 12.1. Отмена заказа клиентом

Клиент может отменить заказ:

* до передачи заказа поставщику;
* до дедлайна сборки.

После дедлайна отмена может быть недоступна или требовать подтверждения поставщика / администратора.

---

## 12.2. Отмена поставщиком

Поставщик может отменить заказ только с причиной:

* товара нет в наличии;
* ошибка в цене;
* проблема с поставкой;
* невозможность доставки;
* другая причина.

---

## 12.3. Возврат

Возврат возможен при:

* товар испорчен;
* товар не соответствует описанию;
* доставлен неправильный товар;
* заказ не доставлен;
* спор решен в пользу клиента.

---

## 12.4. Жалоба

Клиент может пожаловаться на:

* поставщика;
* товар;
* доставку;
* общение;
* нарушение условий.

* клиент может войти по SMS;
* поставщик может войти по SMS;
* поставщик может заполнить профиль;
* поставщик может загрузить логотип и обложку;
* поставщик может выбрать зоны доставки;
* поставщик может задать стоимость доставки;
* поставщик может добавить товар;
* товар автоматически появляется в каталоге;
* клиент может открыть профиль поставщика;
* клиент может выбрать товар;
* клиент видит минимальный объем товара;
* клиент может выбрать количество только по шагу заказа;
* клиент может добавить товары от нескольких поставщиков в корзину;
* корзина разделяет товары по поставщикам;
* система считает итог по каждому поставщику;
* система считает общий итог корзины;
* клиент может оформить заказ;
* клиент может оплатить заказ;
* поставщик видит оплаченный заказ;
* поставщик может изменить статус заказа;
* клиент может написать поставщику;
* клиент может открыть спор;
* администратор может подключиться к спору;
* администратор может скрыть товар;
* администратор может управлять поставщиками, товарами, заказами и жалобами.

# 21. Ключевая логика для разработчиков

Главная особенность Хоми — это не обычный интернет-магазин.

Главная логика:

```text
Товар нельзя купить меньше минимального объема.
Количество можно менять только по заданному шагу.
Поставщик сам выбирает зоны доставки.
Поставщик сам задает стоимость доставки.
Товары публикуются автоматически.
Администратор может скрыть товар после публикации.
Корзина разделяется по поставщикам.
Клиент может общаться с поставщиком напрямую.
Если спор не решен, подключается поддержка Хоми.
```

# 22. Краткое описание ТЗ для команды

Нужно разработать мобильное приложение и web-систему для маркетплейса плановых оптовых закупок продуктов.

В системе есть 3 роли:

* клиент;
* поставщик;
* администратор.

Клиент покупает товары крупными объемами.
Поставщик размещает товары, управляет доставкой и получает оплаченные заказы.
Администратор контролирует поставщиков, товары, заказы, жалобы и споры.

Ключевые требования:

* SMS-авторизация;
* минимальный объем товара;
* единицы измерения: кг, л, шт, упаковка, коробка, мешок;
* шаг заказа;
* минимальная сумма заказа у поставщика;
* корзина по нескольким поставщикам;
* публичный профиль поставщика;
* логотип и обложка поставщика;
* автоматическая публикация товаров;
* скрытие товаров администратором;
* плановая доставка;
* зоны доставки от поставщика;
* стоимость доставки от поставщика;
* онлайн-оплата;
* чат клиента с поставщиком;
* жалобы и споры;
* аналитика просмотров каталога и карточек товаров.
