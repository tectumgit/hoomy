# Wireframes: frontend-first MVP

## Цель документа

Этот файл описывает wireframes для первого этапа разработки Хоми: сначала делаем фронтенд на мок-данных, проверяем логику экранов и пользовательский путь, затем подключаем API и backend.

Старый подход "сразу описать всю финальную систему" для разработки неудобен: он смешивает продуктовую логику, backend-модель, админку, поставщика и клиентское приложение. На старте нужен другой формат:

```text
экраны -> состояния -> мок-данные -> будущие API-контракты
```

## Главные правила wireframes

1. Главный объект покупки — поставщик.
2. Главная страница — лента поставщиков с каруселями товаров.
3. Карточка товара не хранит доставку как свое свойство.
4. Доставка, даты, интервалы, дедлайны и стоимость относятся к поставщику.
5. Корзина всегда группируется по поставщикам.
6. Checkout выбирает дату доставки отдельно по каждому поставщику.
7. На этапе frontend-first все данные берутся из моков.
8. Моки должны быть похожи на будущий API, чтобы потом не переписывать интерфейс.

## Этапы разработки

### Этап 1. Клиентский фронтенд на моках

Цель: собрать мобильный клиентский путь без backend.

Делаем:

* onboarding;
* SMS auth mock;
* главную ленту поставщиков;
* поиск и фильтры;
* каталог поставщика;
* карточку товара;
* выбор количества;
* корзину по поставщикам;
* checkout с мок-оплатой;
* успешный заказ;
* заказы и детали заказа;
* чат-заглушку;
* спор-заглушку;
* профиль.

### Этап 2. Web-кабинет поставщика на моках

Цель: проверить рабочую логику поставщика.

Делаем:

* onboarding поставщика;
* профиль поставщика;
* настройки доставки;
* товары;
* добавление товара;
* заказы;
* сообщения;
* базовую аналитику.

### Этап 3. Web-админка на моках

Цель: проверить модерацию и поддержку.

Делаем:

* dashboard;
* поставщики на проверке;
* карточка поставщика;
* список товаров;
* заказы;
* жалобы;
* споры.

### Этап 4. API и backend

Цель: заменить mock repositories на реальные API.

Правило: компоненты экранов не должны знать, мок это или backend. Экран вызывает один слой:

```text
features/*/api или repositories/*
```

Потом внутри этого слоя моки меняются на HTTP-запросы.


# 3. Кабинет поставщика: web mock

Desktop-first рабочий интерфейс:

```text
Sidebar: Главная | Заказы | Товары | Доставка | Сообщения | Аналитика | Профиль
Topbar: статус поставщика, уведомления, профиль
Content: таблицы, формы, карточки
```

Мобильная адаптация web-кабинета для MVP не приоритетна, но экран не должен ломаться на 390 px.

# 7. Acceptance criteria frontend-first

Клиентский прототип считается готовым, если:

1. Можно пройти путь от onboarding до mock-заказа.
2. Главная показывает поставщиков из моков.
3. Каталог поставщика показывает товары только выбранного поставщика.
4. Количество товара меняется по `minQuantity` и `orderStep`.
5. Корзина группирует товары по `supplierId`.
6. Checkout недоступен, если не выполнен минимум поставщика.
7. Checkout выбирает дату доставки отдельно по каждому поставщику.
8. Mock payment создает order с supplier suborders.
9. Заказ отображается в истории.
10. В UI нет доставки как свойства товара.

Поставщик frontend mock готов, если:

1. Можно добавить товар в mock state.
2. Можно настроить доставку поставщика.
3. Можно увидеть mock-заказы.
4. Можно менять статус заказа локально.

Админка frontend mock готова, если:

1. Можно открыть поставщика на проверке.
2. Можно скрыть товар в mock state.
3. Можно открыть заказ с подзаказами.
4. Можно открыть спор и выбрать mock-решение.
