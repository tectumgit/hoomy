# Vibe Coding Brief: Hoomy / Хоми

Этот файл можно давать AI-разработчику как главный краткий контекст перед генерацией кода.

## Что нужно построить

Построй MVP Hoomy: мобильный маркетплейс плановых закупок продуктов и товаров большими объемами напрямую у поставщиков.

Роли:

* customer — покупает товары;
* supplier_user — управляет поставщиком, товарами, доставкой и заказами;
* admin — проверяет поставщиков, смотрит товары, заказы, жалобы и споры;
* super_admin — системные настройки и комиссии.

## Главная продуктовая логика

1. Главный объект покупки — поставщик.
2. На главной клиент видит ленту поставщиков с каруселями товаров.
3. При клике на поставщика открывается полный каталог поставщика.
4. Товар имеет цену, единицу, минимальный объем, шаг заказа и остаток.
5. Доставка не является свойством товара.
6. Доставка принадлежит поставщику: зоны, дни, интервалы, дедлайн, стоимость, бесплатная доставка от суммы.
7. Корзина группируется по поставщикам.
8. Если поставщиков несколько, checkout создает общий заказ и отдельные подзаказы по поставщикам.
9. Клиент выбирает дату доставки отдельно по каждому поставщику.
10. Товары публикуются автоматически, но администратор может скрыть или заблокировать товар.

## Стек

Используй:

```text
Expo + React Native + TypeScript для mobile
Next.js App Router + TypeScript для supplier/admin web
Next.js route handlers для API MVP
PostgreSQL + Prisma для базы
Zod для validation schemas
TanStack Query для server state
React Hook Form для форм
Tailwind + shadcn/ui для web
pnpm workspaces + Turborepo для монорепозитория
```

## Структура

```text
apps/mobile
apps/web
packages/db
packages/shared
packages/config
```

В `packages/shared` держать:

* Zod-схемы;
* enum-ы;
* типы API;
* бизнес-константы;
* permissions.

В `packages/db` держать:

* Prisma schema;
* migrations;
* Prisma client.

В `apps/web/src/modules` держать бизнес-логику. Route handlers не должны содержать сложную логику.

## Нельзя делать

* Не делать товару поле `deliveryDate` как основную бизнес-модель.
* Не смешивать товары разных поставщиков в один плоский заказ.
* Не давать оформить заказ, если не достигнута минимальная сумма поставщика.
* Не давать выбрать количество меньше минимума.
* Не давать выбрать количество не по шагу.
* Не использовать float для денег.
* Не писать бизнес-логику прямо в React-компонентах.
* Не копировать визуальный стиль, цвета, тексты и иконки Яндекса.

## Правила количества

```text
quantity >= minQuantity
quantity follows orderStep
quantity <= stockQuantity
```

Примеры:

```text
Картофель: мин. 10 кг, шаг 5 кг -> 10, 15, 20
Трюфели: мин. 0.1 кг, шаг 0.1 кг -> 0.1, 0.2, 0.3
Молоко: мин. 6 л, шаг 1 л -> 6, 7, 8
Яйца: мин. 30 шт, шаг 30 шт -> 30, 60, 90
```

## UI-правила

* Интерфейс светлый, современный, быстрый.
* Основной брендовый цвет: `#FF6500`.
* Вторичный свежий цвет: `#45D88A`.
* Поставщик должен выглядеть надежно: логотип, обложка, рейтинг, отзывы, условия доставки.
* В карточке товара сразу показывать минимум, шаг, цену и цену минимальной партии.
* Корзина должна явно показывать блок каждого поставщика.
* Если кнопка недоступна, пользователь должен видеть причину.
* Тексты короткие и человеческие.

## Примеры правильных текстов

```text
Минимальный заказ у поставщика — 3 000 ₽. Добавьте еще на 600 ₽.
Количество можно менять только с шагом 5 кг.
Поставщик доставляет по вторникам, четвергам и субботам.
Закажите до 18:00, чтобы получить доставку завтра.
```

## Definition of Done для MVP

MVP готов, когда:

* клиент входит по SMS;
* видит поставщиков;
* открывает каталог поставщика;
* выбирает товар;
* выбирает количество по минимуму и шагу;
* добавляет товары от нескольких поставщиков в корзину;
* видит корзину по поставщикам;
* выбирает дату доставки по каждому поставщику;
* оплачивает заказ;
* поставщик видит свой подзаказ;
* поставщик меняет статус;
* клиент видит статус;
* клиент и поставщик могут переписываться;
* клиент может открыть спор;
* администратор может управлять поставщиками, товарами, заказами и спорами.
